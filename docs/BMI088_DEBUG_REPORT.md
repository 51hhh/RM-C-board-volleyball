# BMI088 "初始化卡死" 问题 — 最终诊断报告

> 结论先行：**根本不存在 BMI088 初始化卡死。板子一直在正常工作。**
> 真正的问题是 **LED 翻转太快肉眼看不见**，被误判成了"卡死"。

---

## 一、最终结论

| 项目 | 实测结果 |
|------|---------|
| BMI088 初始化 | ✅ 第一次调用即成功（`bmi088_init_attempts=0`）|
| BMI088 数据 | ✅ accel[2] ≈ **9.84 m/s²（重力）**，gyro ≈ 0（静止），数值实时变化 |
| SPI1 通信 | ✅ 正常（RXNE 置位、收到真实数据）|
| 系统时钟 | ✅ 168MHz（HSE+PLL 锁定，APB2=84MHz）|
| 主循环 | ✅ 正常运行 |
| LED (PH11) | ✅ 引脚一直在翻转 |

**所有硬件和外设都正常**。问题纯粹是视觉误判。

## 二、真正的根因

`Core/Src/main.c` 主循环里的 LED 闪烁逻辑：

```c
// 修复前
led_cnt ++;
if (led_cnt == 2) {           // 每 2 次循环翻转一次
    led_cnt = 0;
    HAL_GPIO_TogglePin(GPIOH, GPIO_PIN_11);
    ...
}
```

同时主循环里的延时 `HAL_Delay(40)` 在 549 行被**注释掉了**。

后果：
- 主循环没有任何延时 → 以极高频率运行（数百~数千 Hz）
- LED 每 2 次循环就翻转一次 → 翻转频率高达 **数百 Hz**
- 人眼无法分辨 → LED 看起来是**常亮或不亮** → 被误判为"板子卡死在 BMI088 初始化"

注释里写的是 "周期 500ms (250 * 2ms)"，说明**原始设计**是配合 2ms 循环延时 + `led_cnt==250` 来实现 500ms 闪烁；后来延时被删、阈值被改成 2，闪烁就坏了。

## 三、修复方案（已实施并验证）

把 LED 改成**基于系统时间**的心跳，与循环速度彻底解耦：

```c
// 修复后
if (current_time_stamp - led_blink_stamp >= 500) {
    led_blink_stamp = current_time_stamp;
    HAL_GPIO_TogglePin(GPIOH, GPIO_PIN_11);   // 稳定 500ms 可见心跳
}

// UART 队列每次循环都尝试发送（DMA 空闲时）
if (dma_tx_complete == 1) {
    process_uart_queue();
}
```

无论主循环多快/多慢，LED 都稳定 500ms 翻转一次，肉眼可见。

**新增变量**：`uint32_t led_blink_stamp`（心跳时间戳）。

## 四、硬件验证（probe-rs + embedded-debugger MCP）

烧录修复固件后复位运行，实测：

| 地址 | 变量 | 实测值 | 含义 |
|------|------|--------|------|
| 0x20000AEC | accel[2] | 0x411D7D3E ≈ **9.84** | 重力，BMI088 正常 |
| 0x20000B8C | led_blink_stamp | 116000 → 116500 | **每 500ms 递增一次**，心跳正常 |
| 0x40021C14 | GPIOH ODR bit11 | 0 ↔ 1 翻转 | LED 在闪烁 |

→ 主循环正常、BMI088 正常、LED 以 500ms 可见周期闪烁。**问题解决。**

## 五、关键调试方法（本次最有价值的经验）

1. **不要相信"看起来卡死"** —— 用调试器读内存/寄存器看真实状态，而不是靠现象猜。

2. **栈回溯 + 寄存器解码定位** —— 通过读 SP 重建调用栈、解码 SPI/RCC/GPIO 寄存器，证明了：
   - 时钟正确（RCC_CFGR）
   - SPI 配置正确（SPI1_CR1=0x037F）
   - BMI088 返回真实数据（accel=重力）

3. **诊断变量 + IEEE754 解码** —— 读取 `accel[2]` 的原始 4 字节 `0x411D7D3E` 解码为 9.84 m/s²，一眼确认 BMI088 在读重力。

4. **probe-rs / MCP 实时读内存** —— 在不停机的情况下采样 `led_blink_stamp`、`accel`、`GPIOH ODR`，看它们随时间变化，证明系统活着。

## 六、之前为什么误判

- 历史文档 `BMI088_FIX_SUMMARY.md` 曾以为"加 HAL_Delay 修复了卡死"——其实那只是**碰巧让循环变慢、LED 翻转变慢到能看见**，掩盖了真正原因，并非真的修了 BMI088。
- 调试早期我也一度怀疑 FPU/SPI 超时/初始化顺序，但逐一用寄存器实测**排除**了：纯净原版（soft-float、原始死循环）也能读到重力，证明那些都不是问题。

## 七、修改的文件

- `Core/Src/main.c`：LED 闪烁逻辑改为基于 `HAL_GetTick()` 的 500ms 时间心跳；新增 `led_blink_stamp` 变量；UART 队列处理移出 LED 计数判断。

## 八、可选的后续改进（非必须）

1. **启用硬件 FPU**（`CMakeLists.txt` 取消 `-mfloat-abi=hard -mfpu=fpv4-sp-d16` 注释）：当前是软件浮点，对于每个循环都跑卡尔曼滤波 + 4×PID + 麦轮解算的实时控制系统，硬件 FPU 能大幅提速、提升控制品质。（已验证可正常编译运行，但与本次"卡死"问题无关。）
2. **给主循环加固定节拍**（如基于定时器的固定 1kHz/500Hz 控制周期），让控制更稳定。
3. **SPI 读写加超时返回**（`BMI088_read_write_byte` 用 100ms 超时而非 1000ms），万一传感器真的掉线也不会死等。
